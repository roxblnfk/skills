# Built-in trusted-vendors list for `llm/skills`.
#
# A package whose Composer name matches any of these patterns is allowed to
# ship AI skills into a consumer project without an explicit opt-in. The goal
# is purely safety: AI skills are Markdown instructions executed by an agent,
# so a malicious package could ship a prompt-injection payload. The trust
# list narrows the attack surface to maintainers we are willing to vouch for.
#
# Pattern syntax (see \LLM\Skills\Config\VendorPattern):
#   - `vendor/*`       — every package in the vendor namespace
#   - `vendor/package` — exact package match
#
# Maintainers extend this list by PR. End users override locally via:
#
#     "extra": {
#       "skills": {
#         "trusted": ["my-org/*", "acme/skills-pro"],
#         "trusted-replace": true
#       }
#     }
#
# Format: one pattern per line. Blank lines and lines starting with `#` are
# ignored. Surrounding whitespace is trimmed. Entries are kept in alphabetical
# order — please preserve that when adding a new line.
#
# How to get your vendor added
# ----------------------------
# Open a Pull Request that adds your pattern to the list below. Because there
# is no objective "trust" metric, inclusion is ultimately a judgement call,
# and the bar we apply is community visibility: a vendor that is recognisable
# to PHP developers working with AI tooling is welcome to submit a PR.
#
# If your vendor is newer or less widely known, that is not a blocker — ask
# one or more well-known maintainers from the ecosystem to vouch for you on
# the PR (an approving review is enough). Endorsement from people the
# wider community already trusts is what lets us extend the list responsibly.
#
# Prefer the narrowest pattern that fits your distribution. If only one or
# two packages from your namespace ship skills, list them explicitly
# (`vendor/package`) rather than claiming the whole `vendor/*` namespace —
# this keeps the attack surface small and makes future audits easier.
